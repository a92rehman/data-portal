# Data Portal

## What This Is

Internal data request management system. Team leads submit data requests, analysts manage and complete them. Includes Power BI dashboard embedding, AI chat (InsightFlow), and metric definitions.

## Quick Start

```bash
npm install
npm run dev          # Dev server on port 5000
npm run build        # Production build
npm start            # Run production build
npm run check        # TypeScript type check
npm run db:push      # Push Drizzle schema to DB
```

## Architecture

Monorepo with three packages sharing a single `package.json`:

```
client/src/          React 18 + TypeScript frontend (Vite)
server/              Express.js backend (ESM, tsx runner)
shared/              Shared schema and types (Drizzle + Zod)
```

**Frontend** (`client/src/`):
- **Router**: Wouter (not React Router)
- **State**: TanStack Query (server state), no client state lib
- **UI**: shadcn/ui (Radix primitives) + Tailwind CSS
- **Forms**: React Hook Form + Zod validation
- **Path aliases**: `@/` = `client/src/`, `@shared/` = `shared/`

**Backend** (`server/`):
- **Framework**: Express.js with ESM (`"type": "module"`)
- **ORM**: Drizzle with Neon serverless PostgreSQL
- **Auth**: Passport.js local strategy, session-based (connect-pg-simple)
- **Realtime**: WebSocket (ws) for notifications
- **Entry**: `server/index.ts` bootstraps migrations, routes, and Vite dev server

**Database**: Neon serverless PostgreSQL via `@neondatabase/serverless`
- Schema defined in `shared/schema.ts` (single source of truth)
- Migrations in `migrations/` (raw SQL, run at startup)
- Drizzle Kit for schema push: `npm run db:push`

## Key Files

| File | Purpose |
|------|---------|
| `shared/schema.ts` | All DB tables, enums, relations, Zod schemas, TypeScript types |
| `server/routes.ts` | All API routes (~2000 lines, the main API surface) |
| `server/storage.ts` | Data access layer (CRUD operations via Drizzle) |
| `server/auth.ts` | Passport setup, password hashing (scrypt), session config |
| `server/index.ts` | Server bootstrap: migrations, routes, Vite, error handling |
| `server/insightflow.ts` | Proxy to InsightFlow AI (Cloud Run SSE) |
| `server/powerbiService.ts` | Power BI embed token generation and visual data extraction |
| `client/src/App.tsx` | Root component with routing |
| `client/src/pages/` | Page components (one per route) |
| `client/src/components/` | Reusable components (app-specific + shadcn/ui) |

## User Roles

| Role | Code | Can Do |
|------|------|--------|
| Data Requester | `requester` | Submit requests, view own requests |
| Data Lead | `team_lead` | Full access: review, assign, manage team, analytics |
| Data Analyst | `analyst` | View assigned requests, update status, manage tasks |

## Conventions

- **Imports**: Use `@shared/schema` for shared types, `@/` for client-relative imports
- **API routes**: All under `/api/`, authenticated via `isAuthenticated` middleware
- **IDs**: UUID strings generated by PostgreSQL (`gen_random_uuid()`)
- **Dates**: JavaScript Date objects, stored as `timestamp` in Postgres
- **Booleans in DB**: Some columns use `varchar` with `"true"`/`"false"` (legacy pattern in blockers, notifications)
- **Error handling**: API routes return `{ message: string }` on error

## Environment Variables

Required:
- `DATABASE_URL` - Neon PostgreSQL connection string
- `SESSION_SECRET` - Express session secret

Optional:
- `BREVO_API_KEY` - Server-side email notifications
- `OPENAI_API_KEY` - AI insights and chat
- `INSIGHTFLOW_URL` - InsightFlow AI service URL
- `POWERBI_*` - Power BI embed configuration (see `docs/powerbi/`)
- `PORT` - Server port (default: 5000)

## Gotchas

- Power BI embed token route (`/api/powerbi/embed-token/:reportId`) is intentionally unauthenticated (registered before auth middleware)
- Vite HMR is disabled (`hmr: false`) because it interferes with Power BI iframe rendering
- The `data_analyst` role in the enum is legacy; `analyst` is the current role
- `server/index.ts` runs raw SQL migrations at startup (not Drizzle migrations)
